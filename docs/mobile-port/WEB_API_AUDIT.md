# Web API Audit — Web-only APIs → Expo/RN Replacements

Every browser/Next-specific API the web app depends on, where it's used, the RN/Expo replacement, and a severity (trivial / moderate / hard) reflecting porting effort. A styling recommendation follows the table.

## Severity rollup

- **hard** (plan early, biggest cost): GSAP→Reanimated breadth; Tailwind/liquid-glass→NativeWind+expo-blur; the entire push pipeline; Web Audio tones; LiveKit RN/WebRTC; `getUserMedia`/MediaRecorder audio; the `document.*`/`window.*` DOM/global surface in CallContext and modals.
- **moderate**: navigation, images, fonts, localStorage→AsyncStorage/SecureStore, View Transitions, cookie→Bearer + WebSocket base URL.
- **trivial**: next/link, sessionStorage, clipboard, share, TTS.

## Styling recommendation: NativeWind v4 + expo-blur (not hand-rolled StyleSheet)

**Recommendation: NativeWind v4 + expo-blur, NOT a hand-rolled StyleSheet + theme object.**

**Rationale.** 51 `.tsx` files author with `className` and the design system is small and token-driven (12 color tokens in `@theme` at `src/app/globals.css`). NativeWind v4 keeps `className` largely intact and maps the CSS variables to a `tailwind.config` theme, so the port is mostly mechanical (1:1 utility translation) rather than rewriting every component into StyleSheet objects.

**Concretely:**

1. **Tokens & dark mode.** Move the `:root`/`.light` token values from `globals.css` into `tailwind.config` `theme.extend.colors`, and drive dark/light with NativeWind's `dark:` variant + a color-scheme provider (rather than the `.light` class + CSS vars). NativeWind v4 supports CSS-var theming, but a plain config theme is more robust on native.

2. **The Liquid Glass look is the hard part** — RN has no `backdrop-filter`, no `inset` box-shadow, and limited box-shadow on Android, so `className` alone cannot reproduce `.liquid-glass`. Build a single reusable **`<LiquidGlass>`** component:
   - an **expo-blur `<BlurView intensity={24} tint={scheme}>`** background layer,
   - an absolutely-positioned semi-transparent **border View** (rgba white/black to match `--color-border`),
   - a drop shadow via `shadowColor/shadowOpacity/shadowRadius` (iOS) + `elevation` (Android),
   - the inset highlight (`inset 0 1px 0 rgba(255,255,255,.2)`) faked with a thin top-edge gradient/View.
   Replace all 30 `liquid-glass`/`liquid-glass-card` usages with this component (or a NativeWind `className` styling its inner content).

3. **Animated utilities** (`nav-active-glass`, `liquid-status-halo`, `story-bg`, glows) cannot be CSS keyframes — implement as **react-native-reanimated** looped animations colocated with the component.

4. **`text-glow`/`accent-glow`** text-shadow → RN `textShadowColor`/`textShadowRadius` style props (works, weaker than CSS).

5. **Theme toggle** → `useColorScheme` from NativeWind, persisting the choice to **AsyncStorage** (replacing localStorage `conjuntos_theme`).

**Net:** NativeWind preserves authoring velocity for ~95% of utility classes; the ~5% that depend on blur/shadow/keyframes are encapsulated in a handful of native primitives (expo-blur + Reanimated). A pure StyleSheet+theme-object approach would also work but multiplies the migration cost across 51 files for no real benefit, since the glass/animation problems must be solved with native components either way.
